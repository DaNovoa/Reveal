Introducción a los Patrones de Diseño

Los patrones de diseño son soluciones probadas y reutilizables para problemas comunes de diseño de software.

Utilidad en la resolución de problemas:

  • Ofrecen soluciones eficaces para desafíos de diseño.
  • Promueven la reutilización de soluciones probadas.
  • Facilitan la estandarización y mantenibilidad del código.

Categorías de Patrones de Diseño

  1. Patrones Creacionales
  2. Patrones Estructurales
  3. Patrones de Comportamiento

Patrones Creacionales

Estos patrones se centran en cómo se crean objetos. Proporcionan formas flexibles y reutilizables de crear objetos sin especificar sus clases concretas.

Patrones Estructurales

Los patrones estructurales se centran en cómo los objetos y las clases se combinan para formar estructuras más grandes. Estos patrones se utilizan para construir sistemas de objetos más grandes y proporcionar flexibilidad en la composición de objetos.

Patrones de Comportamiento

Los patrones de comportamiento se centran en la comunicación entre objetos y cómo se distribuye la responsabilidad entre ellos. Estos patrones ayudan a definir cómo los objetos interactúan y colaboran para completar tareas.

Patrón Creacional - Singleton

Singleton

El patrón Singleton es un patrón de diseño creacional que se utiliza para garantizar que una clase tenga una única instancia y proporcionar un punto de acceso global a esa instancia. Su objetivo principal es restringir la creación de objetos de una clase a una sola instancia, lo que es útil en situaciones donde necesitas tener un único punto de control para una determinada funcionalidad o recurso.

Características clave del patrón Singleton:

  • Una única instancia: Garantiza que solo exista una instancia de la clase Singleton en todo el programa.
  • Acceso global: Proporciona un método estático (a menudo llamado getInstance()) que permite acceder a la instancia única desde cualquier parte del código.
  • Lazy initialization: La instancia se crea solo cuando se solicita por primera vez, en lugar de en el momento de la carga de la clase.
  • Control sobre la instancia: Puede haber una lógica adicional para garantizar que la instancia sea creada de manera segura y controlada.
  • Thread-safety: A menudo se implementa con consideraciones de concurrencia para garantizar que la instancia única sea segura para usar en entornos multiproceso.

Patrón Creacional - Factory Method

Factory Method

El patrón Factory Method es un patrón de diseño creacional que se utiliza para crear objetos sin especificar la clase concreta del objeto que se creará. En lugar de crear objetos directamente utilizando un constructor, el Factory Method define una interfaz para crear objetos, pero delega la decisión de qué clase concreta crear a las subclases o implementaciones concretas. Esto permite que una clase delegue la responsabilidad de instanciar objetos a sus subclases, lo que promueve la flexibilidad y la extensibilidad del código.

Patrónes Estructurales

Adapter (Adaptador):

El patrón Adapter es un patrón de diseño estructural que se utiliza para permitir que interfaces incompatibles o no coincidentes trabajen juntas. Su objetivo principal es actuar como un puente entre dos interfaces que no pueden trabajar directamente entre sí debido a diferencias en su estructura o funcionalidad.

Bridge (Puente)

Divide una abstracción de su implementación, de modo que ambas puedan variar de manera independiente.

Composite (Compuesto)

Permite que los objetos se compongan en estructuras de árbol para representar jerarquías parte-todo. Los clientes pueden tratar tanto a los objetos individuales como a las composiciones de manera uniforme.

Decorator (Decorador)

Agrega funcionalidad adicional a los objetos dinámicamente. Proporciona una forma flexible de extender las capacidades de las clases.

Facade (Fachada)

Proporciona una interfaz unificada para un conjunto de interfaces en un subsistema. Simplifica la interacción del cliente con el sistema.

Flyweight (Peso Ligero)

Minimiza el uso de la memoria o el almacenamiento compartiendo tanto como sea posible con objetos relacionados. Útil cuando se trabaja con grandes cantidades de objetos similares.

Proxy (Proxy)

Proporciona un sustituto o representante de otro objeto para controlar el acceso a él. Útil para la creación de objetos costosos bajo demanda o para agregar una capa de control.

Patrónes Estructurales

Composite

El patrón Composite es un patrón de diseño estructural que se utiliza para tratar objetos individuales y composiciones de objetos de la misma manera. Su objetivo principal es componer objetos en estructuras de árbol para representar jerarquías de objetos "todo-parte" o estructuras anidadas. Esto permite que los clientes traten tanto a objetos individuales como a composiciones de objetos de manera uniforme, lo que simplifica el código y proporciona flexibilidad en la manipulación de estructuras complejas.

Patrónes de Comportamiento

Observer

El patrón Observer es un patrón de diseño de comportamiento que se utiliza para definir una relación de uno a muchos entre objetos, de modo que cuando un objeto cambia su estado, todos sus observadores sean notificados y actualizados automáticamente. Este patrón se basa en el principio de desacoplar el sujeto (objeto que es observado) de sus observadores, lo que permite una mayor modularidad y flexibilidad en el diseño del software.

Patrónes de Comportamiento

Strategy

El patrón Strategy es un patrón de diseño de comportamiento que permite definir una familia de algoritmos, encapsular cada uno de ellos y hacer que sean intercambiables. Esto significa que el patrón Strategy permite cambiar el comportamiento de un objeto en tiempo de ejecución sin alterar su estructura.

Command (Comando)

Encapsula una solicitud como un objeto, lo que le permite parametrizar clientes con colas, solicitudes y operaciones. También permite soportar operaciones reversibles.

State (Estado)

Permite que un objeto cambie su comportamiento cuando su estado interno cambia. Parece que el objeto cambia su clase.

Chain of Responsibility (Cadena de Responsabilidad)

Pasa las solicitudes a lo largo de una cadena de manejadores. Cada manejador decide si procesa la solicitud o la pasa al siguiente manejador en la cadena.

Memento (Recuerdo)

Captura y externaliza un estado interno de un objeto, lo que permite restaurar el objeto a ese estado en el futuro.

Interpreter (Intérprete)

Define una gramática para un lenguaje y proporciona una interpretación para sentencias en el lenguaje.

Visitor (Visitante)

Representa una operación a realizar en los elementos de una estructura de objetos. Permite definir una nueva operación sin cambiar las clases de los elementos en los que opera.

Template Method (Método Plantilla)

Define el esqueleto de un algoritmo en la superclase pero permite que las subclases sobrescriban pasos específicos del algoritmo sin cambiar su estructura general.

Conclusión

Para Mi:

Los patrones de diseño son soluciones probadas y reutilizables que abordan problemas comunes de diseño en el desarrollo de software. Estas herramientas, como Singleton, Factory Method y otros, mejoran la eficiencia, la mantenibilidad y la escalabilidad del código. Su importancia radica en la reutilización, estandarización y abstracción de problemas complejos, lo que facilita el desarrollo de sistemas robustos y adaptables.
